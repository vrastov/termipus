# Участие в проекте

## Рабочий процесс

1. Найти или создать Issue в GitLab
2. Создать ветку от актуального `main`: `feature/<имя>/#<номер issue>`
3. Реализовать задачу, покрыть тестами (≥ 80% на новом коде)
4. Создать MR в Draft, дождаться CI
5. Исправить замечания SonarCloud
6. Схлопнуть коммиты в один (`git rebase -i origin/main`)
7. Снять Draft, назначить ревьюера
8. После апрува — merge

## Соглашения

### Коммиты

Формат: `#N <тип>[область]: <краткое описание>`
Тип обязателен, область — опциональна. Типы: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`

```
#5 docs: добавить документацию проекта

Подробное описание что сделано и почему.
```

### Java

- Google Java Style Guide: отступ 2 пробела, длина строки 100 символов
- Javadoc обязателен на всех public классах и методах
- Фигурные скобки обязательны даже для однострочных `if`/`for`/`while`
- Строки только с явным `StandardCharsets.UTF_8`
- Не ловить `Exception` — только конкретные исключения
- Перед коммитом: `mvn sortpom:sort` — сортировка зависимостей и плагинов в pom.xml (проверяется в CI через `checkPomXml`)

### TypeScript

- Строгий режим (`strict: true`), без `any`
- Тесты рядом с файлом: `socket.ts` → `socket.test.ts`

## CI

Pipeline запускается автоматически на каждый push и MR:
- `build` — сборка и тесты обоих модулей
- `sonar` — анализ кода (только `main` и MR)

Quality Gate блокирует merge при покрытии < 80% или критических замечаниях.
